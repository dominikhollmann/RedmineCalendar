# Implementation Plan: Booking Guard Warnings

**Branch**: `042-booking-guard-warnings` | **Date**: 2026-06-17 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/042-booking-guard-warnings/spec.md`

## Summary

Two soft-warning dialogs intercept time-entry mutations (create, edit, delete, drag-move) that touch sensitive periods: bookings on a future calendar date (user-error guard) and bookings that would change what a resubmitted weekly report would show (deadline guard). Both reuse the existing `showConfirmDialog` pattern from feature 040. All guard logic lives in a new pure module `js/booking-guard.js` that is independently unit-testable and called from three integration points in the existing codebase.

## Technical Context

**Language/Version**: JavaScript ES2022 (vanilla ES modules, no transpilation)

**Primary Dependencies**: FullCalendar v6 (CDN, existing), MSAL.js v2 (CDN, existing, unchanged)

**Storage**: `config.json` — new optional `bookingDeadline` block (admin-managed, server-side). No new localStorage keys.

**Testing**: Vitest (unit — guard logic), Playwright (UI — dialog flows)

**Target Platform**: Modern browser (desktop-first, mobile supported)

**Project Type**: Static SPA

**Performance Goals**: Guard check is synchronous for the deadline (no network call); future-date check is date comparison only. Both complete in < 1 ms.

**Constraints**: No new npm runtime dependencies. `booking-guard.js` must stay ≤ 500 effective LOC (module-size SQI gate).

**Scale/Scope**: Single-user per browser session; guard fires on every qualifying mutation.

## Phase 0 — Research

See [`research.md`](research.md). All decisions resolved; no NEEDS CLARIFICATION remains.

Key decisions:

- Reuse `showConfirmDialog` (same as closed-ticket gate)
- New module `js/booking-guard.js` (pure logic layer, independently testable)
- null `startTime` → defaults to 00:00 for deadline comparison (conservative)
- Deadline moment computed client-side as most recent past occurrence of configured weekday+hour
- Exemptions (holidayTicket/vacationTicket) apply to future-date warning only
- Bulk-delete: single deadline dialog covers all affected entries

### `js/booking-guard.js` — module design

Three exported functions:

```
runSaveGuards(opts)                              → Promise<boolean>
  opts: { date, startTime, originalDate, originalStartTime, issueId, cfg }
  Runs: 1. future-date guard  2. deadline guard (create or edit)

runDeleteGuard(date, startTime, cfg)             → Promise<boolean>
  Runs: deadline guard (delete operation)

deadlineTriggeredForMove(oDate, oTime, nDate, nTime, cfg) → boolean
  Synchronous — used by eventDrop/eventResize before showing dialog
```

Internal helpers (not exported): `lastDeadlineBefore`, `toDatetime`, `deadlineTriggered`, `isExempt`.

### Integration points (minimal diffs)

**`time-entry-form.js` — `doSave()`** (replace the current closed-ticket check position):

```
// After closed-ticket check passes (or for non-closed tickets):
const cfg = getCentralConfigSync();
const ok = await runSaveGuards({
  date: payload.spentOn,
  startTime: payload.startTime,
  originalDate: _currentEntry?.date ?? null,
  originalStartTime: _currentEntry?.startTime ?? null,
  issueId: _selectedIssue?.id ?? null,
  cfg,
});
if (!ok) { setSaveButtonBusy(false); return; }
await _executeSave(payload);
```

**`time-entry-form.js` — `onDeleteClick()`** (before the existing `openConfirmOverlay` call):

```
const cfg = getCentralConfigSync();
const ok = await runDeleteGuard(_currentEntry.date, _currentEntry.startTime, cfg);
if (!ok) return;
openConfirmOverlay(async () => { /* existing delete logic */ });
```

**`calendar.js` — `eventDrop()`** (after the existing `_checkClosedAndConfirm`, same pattern):

```
const cfg = getCentralConfigSync();
if (deadlineTriggeredForMove(entry.date, entry.startTime, newDate, newTime, cfg)) {
  const proceed = await new Promise(resolve =>
    showConfirmDialog({ title: t('bookingGuard.deadlineTitle'),
                        message: t('bookingGuard.deadlineBody'),
                        onConfirm: () => resolve(true), onCancel: () => resolve(false) })
  );
  if (!proceed) { info.revert(); return; }
}
```

**`calendar.js` — `eventResize()`**: same pattern with `origDate = newDate = entry.date`, `origTime = newTime = entry.startTime` (start is unchanged by resize).

**`entry-commands.js` — `_handleDelete()`** (before `showDeleteConfirm`):

```
const cfg = getCentralConfigSync();
const anyInPeriod = _deletableItems().some(({ entry }) =>
  deadlineTriggeredForMove(entry.date, entry.startTime, entry.date, entry.startTime, cfg)
);
// (reuse deadlineTriggeredForMove with origDate=newDate: equivalent to delete check)
if (anyInPeriod) {
  const proceed = await new Promise(resolve =>
    showConfirmDialog({ ..., onConfirm: () => resolve(true), onCancel: () => resolve(false) })
  );
  if (!proceed) return;
}
showDeleteConfirm(() => { /* existing bulk delete */ });
```

### Unit test plan (`tests/unit/booking-guard.test.js`)

Coverage target: ≥ 95% line coverage on `js/booking-guard.js`.

| Test group           | Cases                                                                                                                    |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `lastDeadlineBefore` | deadline disabled → null; past cutoff → correct Date; before cutoff today → go back 7d; exact cutoff moment → go back 7d |
| `toDatetime`         | null time → 00:00; valid HH:MM → correct Date                                                                            |
| `deadlineTriggered`  | create: new start ≤ deadline → true; new start > deadline → false                                                        |
| `deadlineTriggered`  | edit: orig ≤ deadline → true; new ≤ deadline → true; both > deadline → false                                             |
| `deadlineTriggered`  | delete: orig ≤ deadline → true; orig > deadline → false                                                                  |
| `deadlineTriggered`  | at exact deadline moment (≤, inclusive) → true                                                                           |
| `runSaveGuards`      | future date + non-exempt ticket → shows future-date dialog                                                               |
| `runSaveGuards`      | future date + exempt ticket → skips future-date dialog                                                                   |
| `runSaveGuards`      | today's date → no future-date dialog                                                                                     |
| `runSaveGuards`      | past deadline, create → shows deadline dialog                                                                            |
| `runSaveGuards`      | both conditions → future-date dialog first, then deadline                                                                |
| `runDeleteGuard`     | past deadline → shows deadline dialog                                                                                    |
| `runDeleteGuard`     | deadline disabled → returns true immediately                                                                             |
